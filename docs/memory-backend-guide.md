# VR Conversation Memory Integration Guide

## What is implemented

`POST /ask` and `POST /voice_ask` support server-side conversational memory.
The first request may omit `conversation_id`; the API generates one and returns
it. Reuse that ID on later requests so the assistant receives the earlier user
and assistant turns and can answer follow-up questions.

Memory is ephemeral and bounded:

- it is stored only inside the API process;
- it expires after `CHAT_MEMORY_TTL_S` seconds of inactivity;
- the least recently used conversation is removed when
  `CHAT_MEMORY_MAX_CONVERSATIONS` is reached;
- each conversation is trimmed by `CHAT_MEMORY_MAX_MESSAGES` and
  `CHAT_MEMORY_HISTORY_CHARS`;
- an API container restart or deployment clears all conversations.

The client must still send current `scenario_state` and `lab` fields on every
request. Conversation history helps interpret dialogue, but it is not treated
as an authoritative source for the live VR scene.

## Text integration: `/ask`

Start a conversation without an ID:

```http
POST /ask
Authorization: Bearer <INTERNAL_API_KEY>
Content-Type: application/json

{
  "query": "Что такое кипение?",
  "lab": {
    "subject": "physics",
    "grade": 10,
    "lang": "ru",
    "lab_number": 2
  }
}
```

The response contains the generated ID:

```json
{
  "answer": "...",
  "conversation_id": "0e8c0ea8-3ad6-49c4-8678-7f12c33024b2",
  "citations": []
}
```

Send the same ID with the follow-up:

```json
{
  "query": "Почему оно начинается?",
  "conversation_id": "0e8c0ea8-3ad6-49c4-8678-7f12c33024b2",
  "lab": {
    "subject": "physics",
    "grade": 10,
    "lang": "ru",
    "lab_number": 2
  }
}
```

For streaming requests, read `conversation_id` from the final SSE `done` event.

## Voice integration: `/voice_ask`

Send `conversation_id` as a multipart form field alongside the audio file. If
it is omitted, capture the generated value from the JSON response.

```bash
curl -X POST "$API_BASE/voice_ask" \
  -H "Authorization: Bearer $INTERNAL_API_KEY" \
  -F "file=@question.webm" \
  -F "conversation_id=$CONVERSATION_ID" \
  -F "subject=physics" \
  -F "grade=10" \
  -F "lang=ru" \
  -F "lab_number=2"
```

For streaming voice requests, `conversation_id` is present in both the first
SSE `question` event and the final `done` event.

The same ID can be shared between `/ask` and `/voice_ask`, so a student may ask
the first question by text and continue by voice, or the other way around.

## Unity-side state

Keep one ID on the object that owns the active student/lab session. The exact
HTTP library may differ, but the state flow is:

```csharp
private string conversationId;

// Before each text request:
request.conversation_id = conversationId;

// After each successful JSON response or SSE question/done event:
conversationId = response.conversation_id;

// For voice multipart requests:
if (!string.IsNullOrEmpty(conversationId))
    form.AddField("conversation_id", conversationId);

// When the lab/session ends:
await DeleteConversation(conversationId);
conversationId = null;
```

## Recommended VR lifecycle

1. At the start of a student or lab session, clear the locally cached ID.
2. Make the first `/ask` or `/voice_ask` request without an ID.
3. Save the returned `conversation_id` in the active VR session object.
4. Reuse it for every text and voice follow-up in that session.
5. Continue sending fresh scene and lab context with every request.
6. When the student exits or resets the lab, forget the server conversation:

```http
DELETE /v1/conversations/{conversation_id}
Authorization: Bearer <INTERNAL_API_KEY>
```

Do not reuse one conversation ID across students. The current API authenticates
the VR application with a shared internal key, so the ID itself is the session
boundary. Treat it as opaque and use the UUID generated by the backend.

## Backend flow

1. `app/api/routes.py` obtains or generates the conversation ID.
2. `app/services/memory.py` loads recent history and appends the new user turn.
3. `app/services/llm.py` adds recent dialogue text to the retrieval query so an
   ambiguous follow-up can retrieve material about the original topic.
4. The full trimmed user/assistant history is sent to the OpenAI Responses API.
5. A successful answer is added to memory. Failed requests are not stored.

`POST /v1/chat/completions` remains client-managed: its caller sends the full
`messages` array on every request and does not need `conversation_id`.

## Configuration

| Setting | Default | Meaning |
| --- | ---: | --- |
| `CHAT_MEMORY_MAX_CONVERSATIONS` | 1024 | Maximum active server-side conversations per API process. |
| `CHAT_MEMORY_TTL_S` | 7200 | Inactivity lifetime in seconds. |
| `CHAT_MEMORY_MAX_MESSAGES` | 16 | Maximum recent messages retained per conversation. |
| `CHAT_MEMORY_HISTORY_CHARS` | 6000 | Character budget for model conversation history. |
| `CHAT_MEMORY_RETRIEVAL_CONTEXT_CHARS` | 1200 | Recent dialogue text added to follow-up retrieval. |

Set `CHAT_MEMORY_MAX_CONVERSATIONS=0` or `CHAT_MEMORY_TTL_S=0` to disable
server-side retention. In a multi-replica deployment, use sticky routing or
replace the in-process store with a shared database/cache before scaling the API
beyond one worker.
